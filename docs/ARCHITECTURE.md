# Architecture Overview

## Repository Purpose

**shared-assets** is a **centralized library repository** that provides reusable components for GitHub-based automation and infrastructure provisioning. Rather than managing Azure resources directly (which would require a Terraform state), this repository serves as a **template and distribution hub** for:

- Standard CI/CD workflows (GitHub Actions YAML templates)
- AI-powered GitHub Agentic Workflows for issue-driven automation
- Custom Copilot agents for specialized tasks
- Developer prompts and instructions
- DevContainer configuration for consistent local environments

The repository is designed to be **synced into consuming repositories** (ALZ provisioning, GitHub config, workload repos, etc.) via the `sync-shared-assets` agentic workflow, enabling teams across the organization to share common automation patterns without duplicating code.

## Component Architecture: The Three-Tier Agent Model

**shared-assets** implements a three-tier agent architecture that bridges local development and cloud execution:

### 1. [Local agent] `/generate-documentation` (`.github/prompts/generate-documentation.prompt.md`, `.github/agents/documentation-conductor.agent.md`)

**Invocation:** VS Code command: `/generate-documentation`

**Purpose:** Gather migration target details (organization, repository name) and orchestrate the full documentation generation workflow.

**How it works:**
1. User runs `/generate-documentation` in VS Code
2. Agent asks for target org and target repo
3. Agent invokes the Documentation Conductor with `TARGET_ORG` and `TARGET_REPO` context
4. Conductor delegates specialized tasks to downstream agents

**Runtime:** Local IDE (VS Code devcontainer)

---

### 2. [Agentic Workflow] `alz-vending-dispatcher` (`.github/workflows/alz-vending-dispatcher.md`, `.github/workflows/alz-vending-dispatcher.lock.yml`)

**Definition File:** `.github/workflows/alz-vending-dispatcher.md`  
**Compiled File:** `.github/workflows/alz-vending-dispatcher.lock.yml` (auto-generated by `gh aw compile`)

**Trigger:** Issue opened or closed with `alz-vending` label in `alz-subscriptions` or calling repos

**Purpose:** Act as a dispatcher that assigns the `alz-vending` Copilot coding agent to ALZ provisioning issues and orchestrates cross-repository handoff to `github-config` when the ALZ issue closes.

**How it works:**

1. **Issue opened event:** 
   - Dispatcher reads the issue labels
   - If `alz-vending` label is present, dispatches the `alz-vending` Copilot coding agent
   - Agent receives issue details and begins solution development

2. **Issue closed event:**
   - Dispatcher notifies the requester that their landing zone has been provisioned
   - Extracts configuration details from the closed issue
   - Creates a new issue in `<YOUR_GITHUB_ORG>/github-config` to trigger workload repository provisioning
   - Hands off to the GitHub Config dispatcher

**Authentication:** 
- Uses `GH_AW_AGENT_TOKEN` (fine-grained PAT with Issues: R&W scope)
- Token must have access to both triggering repo and `github-config` target repo

**Engine:** Copilot (Claude)

**Safe Outputs:** 
- `assign-to-agent` — Assign `alz-vending` agent
- `add-comment` — Notify issue creator
- `create-issue` — Cross-repo issue in `github-config`

---

### 3. [Cloud coding agent] `alz-vending` (`.github/agents/alz-vending.agent.md` in `alz-subscriptions` repo)

**File:** Referenced at `<YOUR_GITHUB_ORG>/alz-subscriptions/.github/agents/alz-vending.agent.md`

**Invocation:** Assigned by the `alz-vending-dispatcher` agentic workflow

**Purpose:** Copilot coding agent specialized in Azure Landing Zone vending — interprets user requirements from the issue, generates the Terraform code or configuration needed to provision a landing zone, and manages the provision-to-close handoff.

**Runtime:** Cloud execution (GitHub Actions environment with Copilot agent runtime)

**Lifecycle:**
1. Dispatcher assigns this agent to an ALZ provisioning issue
2. Agent reads issue details, validates requirements
3. Agent generates Terraform code or infrastructure-as-code
4. Agent creates a pull request with changes
5. When PR is merged and issue closes, dispatcher handles cross-repo notifications

---

### Parallel Agents: Documentation Generation Pipeline

The Documentation Conductor also manages a parallel team of agents for documentation generation:

| Component Type | Component Name | Files | Purpose |
|---|---|---|---|
| [Local agent] | Documentation Conductor | `.github/agents/documentation-conductor.agent.md` | Master orchestrator; validates freshness, delegates specialized tasks |
| [Agentic Workflow] | (None used) | — | Documentation generation doesn't require agentic workflows |
| [Cloud coding agent] | SE: Tech Writer | `.github/agents/se-technical-writer.agent.md` | Handoff receives context; generates polished documentation (Setup, Architecture, README) |

**Documentation Workflow:**
```
User runs: /generate-documentation
     ↓
Documentation Conductor (local agent)
     ├─ Validates artifact freshness
     ├─ Scans codebase (Step 1)
     ├─ Extracts prerequisites (Step 2)
     └─ Delegates to SE: Tech Writer (Steps 3-5)
           ├─ Setup Guide (how-to)
           ├─ Architecture Overview (explanation)
           └─ README.md (summary)
```

## Workflow Orchestration Patterns

### Pattern 1: Issue-Driven Dispatch (ALZ Vending Pipeline)

```
[Developer creates ALZ issue]
        ↓
[alz-vending-dispatcher on:issue:opened]
        ↓
[Dispatcher reads labels]
        ├─ Has 'alz-vending' label? → YES
        │       ↓
        │ [assign_to_agent: alz-vending]
        │       ↓
        │ [alz-vending Copilot agent]
        │ (generates Terraform, creates PR)
        │       ↓
        └─ NO: [noop]

[Developer merges PR, closes issue]
        ↓
[alz-vending-dispatcher on:issue:closed]
        ↓
[Dispatcher reads closed issue details]
        ├─ Has 'alz-vending' label? → YES
        │       ↓
        │ [add_comment: Notify requester]
        │       ↓
        │ [create_issue in github-config]
        │       ↓
        │ [github-config-dispatcher triggered]
        │       ↓
        │ [assign_to_agent: github-config]
        │ (manages GitHub provisioning)
        │       ↓
        └─ NO: [noop]
```

### Pattern 2: Reusable Workflow Invocation (Azure Terraform Deploy)

```
[Consuming repo (e.g., workload repo)]
        ↓
[Workflow calls: azure-terraform-deploy.yml]
  (as reusable workflow)
        ↓
[validate job]: fmt, validate, tflint
        ↓
[security job]: Checkov SARIF scan
        ↓
[plan job]: terraform plan, upload artifact
        ↓
[PR comment]: Show plan preview
        ↓
[apply job] (if main branch):
  ├─ requires: environment: azure-production
  ├─ requires: approval from reviewers
  │     ↓
  │ [terraform apply]
  │ (executes plan)
```

## Authentication Model

### 1. Agentic Workflows: GitHub Fine-Grained PAT

**Token:** `GH_AW_AGENT_TOKEN`

**What it does:**
- Enables agentic workflows to assign agents to issues
- Enables cross-repository issue creation
- Enables reading issue details for context

**Permissions:**
- **Issues** — `Read and Write`
- **Pull Requests** — `Read and Write` (optional, for handoff context)

**Scope:** Organization-level or repository-level (if limited to specific repos)

**Lifetime:** 90 days (recommended; align with your org's secret rotation policy)

**Storage:** GitHub Actions secret `GH_AW_AGENT_TOKEN` (org or repo level)

### 2. Reusable Terraform Workflows: Azure OIDC

**Authentication method:** OIDC (OpenID Connect) with Federated Credentials

**How it works for consuming repos:**

Consuming repositories that call `azure-terraform-deploy.yml` pass Azure client IDs (not secrets):

```yaml
jobs:
  deploy:
    uses: <YOUR_GITHUB_ORG>/shared-assets/.github/workflows/azure-terraform-deploy.yml@main
    with:
      environment: azure-production
    secrets:
      AZURE_CLIENT_ID_PLAN: ${{ secrets.AZURE_CLIENT_ID_PLAN }}
      AZURE_CLIENT_ID_APPLY: ${{ secrets.AZURE_CLIENT_ID_APPLY }}
      AZURE_CLIENT_ID_TFSTATE: ${{ secrets.AZURE_CLIENT_ID_TFSTATE }}
      AZURE_TENANT_ID: ${{ secrets.AZURE_TENANT_ID }}
      AZURE_SUBSCRIPTION_ID: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

**Inside the reusable workflow:**

1. **Plan phase:** Uses `AZURE_CLIENT_ID_PLAN` + OIDC token → Terraform performs read-only operations
2. **Apply phase:** Uses `AZURE_CLIENT_ID_APPLY` + OIDC token → Terraform performs create/modify/delete operations
3. **Backend:** Uses `AZURE_CLIENT_ID_TFSTATE` + OIDC token → Terraform accesses state file in Azure Storage

**Why separate client IDs?**
- **Principle of Least Privilege:** Plan identity only has Reader role; Apply identity has Owner role
- **Security:** Even if plan phase is compromised, attacker cannot modify infrastructure
- **Audit trail:** Plan and apply operations are attributed to different identities

### 3. DevContainer: Local Development

**Authentication:** Manual — Use local Azure CLI login

Inside the devcontainer:
```bash
# Login to Azure
az login

# Your ~/.azure folder is mounted into container
# So you have access to your local Azure credentials
terraform plan  # Uses your logged-in identity
```

## Organization String Migration Pattern

**Source Organization:** `nathlan`  
**Target Organization:** `insight-agentic-platform-project`

When migrating shared-assets to a new organization, every reference to `nathlan` must be updated to `insight-agentic-platform-project`:

### Files requiring updates (**.md and .yml):**

| File | Reference | Update |
|------|-----------|--------|
| `.github/workflows/alz-vending-dispatcher.md` | `target-repo: "nathlan/github-config"` | `target-repo: "insight-agentic-platform-project/github-config"` |
| `.github/workflows/alz-vending-dispatcher.md` | References to `nathlan/alz-subscriptions` | Replace with `insight-agentic-platform-project/alz-subscriptions` |
| `.github/workflows/github-config-dispatcher.md` | References to `nathlan` org | Replace with `insight-agentic-platform-project` |
| `.github/workflows/azure-terraform-deploy.yml` | `nathlan/shared-assets` in reference | `insight-agentic-platform-project/shared-assets` |
| `sync/.github/workflows/sync-shared-assets.md` | `nathlan/shared-assets` (source) | `insight-agentic-platform-project/shared-assets` |
| `sync/.github/workflows/grumpy-compliance-officer.md` | `nathlan/shared-standards` | `insight-agentic-platform-project/shared-standards` |
| `sync/.github/agents/grumpy-compliance-officer.agent.md` | `nathlan/shared-standards` | `insight-agentic-platform-project/shared-standards` |

### Files automatically updated (compiled):

The following `.lock.yml` files are **auto-generated** from `.md` files and must NOT be edited manually:

- `.github/workflows/alz-vending-dispatcher.lock.yml`
- `.github/workflows/github-config-dispatcher.lock.yml`

After updating `.md` files, regenerate these by running:
```bash
gh aw compile
```

## DevContainer Architecture

The DevContainer provides a complete, isolated environment for working with this repository:

**Technology Stack:**
- Base: Ubuntu 24.04 LTS
- Container Runtime: Docker (Docker-in-Docker for running MCP servers)
- Infrastructure as Code: Terraform + TFLint
- Scripting: Python 3.11 + pip, Node.js LTS + npm
- VCS: Git + GitHub CLI (gh) + GitHub Agentic Workflows (gh-aw) extension
- AI: Copilot (via VS Code extensions)

**Key Features:**

1. **Terraform Development:**
   - Terraform language server enabled
   - Auto-format on save
   - TFLint for linting
   - Plugin cache directory at `.terraform.d/`

2. **MCP Server Support:**
   - GitHub MCP Server (read/write GitHub resources)
   - Terraform MCP Server (provider/module information)
   - Agentic Workflows MCP Server (via gh-aw)

3. **Local Development Mounts:**
   - `~/.azure/` → `/home/vscode/.azure/` — Azure CLI credentials
   - `~/.ssh/` → `/home/vscode/.ssh/` (read-only) — SSH key access for Git

4. **Post-Create Workflow:**
   - Runs `.devcontainer/setup.sh` to initialize environment
   - Configures git safe directory (required for volume mounts)

## Project Lifecycle

**Development Cycle:**

```
1. [Developer needs to extend shared-assets]
   ↓
2. [Opens repo in DevContainer]
   ├─ Terraform, Python, Node.js, Docker all available
   ├─ GitHub CLI with gh-aw extension pre-installed
   └─ MCP servers configured in .vscode/mcp.json
   ↓
3. [Edits workflow definitions (.md files)]
   ├─ Agentic workflows (.github/workflows/*.md)
   ├─ Agents (.github/agents/*.agent.md)
   └─ Prompts (.github/prompts/*.prompt.md)
   ↓
4. [If editing agentic workflows (.md)]
   ├─ Run: gh aw compile
   └─ Regenerates .lock.yml (compiled GitHub Actions YAML)
   ↓
5. [Commits and pushes to target org]
   ├─ Both .md and .lock.yml files committed
   └─ Consuming repos see compiled workflows
   ↓
6. [End: Consuming repos call shared workflows]
   ├─ Reusable workflows: via `uses:` clause
   ├─ Agentic workflows: auto-dispatched on label triggers
   └─ Agents: assigned to issues by dispatchers
```

## Summary

**shared-assets** is a **library repository** that distributes reusable automation components across the organization using:

- **Dispatcher pattern** — Agentic workflows listen for issue labels and dispatch agents
- **Three-tier agent model** — Local prompts → Agentic workflow orchestration → Cloud agent execution
- **Organization-aware migration** — All references to source org (`nathlan`) must be updated for target org (`insight-agentic-platform-project`)
- **DevContainer-first development** — Terraform, GitHub CLI, and MCP servers pre-configured
- **Zero Terraform state** — This is a library repo; no managed infrastructure, only templates and automation

The repository exemplifies modern GitHub-native automation: using agentic workflows, custom agents, and GitHub Actions to create self-service infrastructure provisioning platforms without managing complex state files.
